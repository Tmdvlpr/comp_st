# 09. Методика обучения моделей (подробный отчёт)

Отчёт описывает, **как именно** в проекте CS Monitor AI обучаются модели выявления аномалий. Источник — `backend/train_and_save_models.py` (+ подготовка данных в `backend/data_loader.py`). Использование обученных моделей в детекции — [06_anomaly_detection.md](06_anomaly_detection.md); аудит метрик/качества — [02_architecture_audit.md](02_architecture_audit.md); план улучшений — [00_implementation_plan.md](00_implementation_plan.md) (фазы 5–7).

---

## 1. Резюме

- **Подход:** для каждого датчика обучается **отдельная регрессионная модель**, предсказывающая его «ожидаемое» значение по другим датчикам того же ГПА и собственной короткой истории. Это модель **физической согласованности**: аномалия = расхождение факта с тем, что «договорились» остальные сигналы и недавнее прошлое.
- **Алгоритм:** CatBoost, функция потерь `RMSEWithUncertainty` (даёт и прогноз, и оценку дисперсии).
- **Данные:** вся история из `raw_data` (или до `--cutoff-date`), 5-минутная сетка, рабочие режимы ГПА (простои отфильтрованы).
- **Валидация:** временно́е разбиение (последние 20% — валидация), early stopping, затем финальное дообучение на 100% данных с подобранным числом итераций.
- **Артефакты:** `models/<station>/<sensor>.cbm` + единый `metadata.json` со всеми метаданными и порогами.
- **Режим:** обучение **однократное** (офлайн), в рантайме модели только загружаются; переобучение не предусмотрено.

---

## 2. Постановка задачи

Для целевого датчика `y` (например `gas_pressure_out_gpa__GPA1`) строится модель:

```
ŷ(t) = f( другие_датчики_ГПА(t),  y(t-1), y(t-2), y(t-3),  скользящие(y) )
```

Модель не классифицирует аномалии напрямую — она учит **норму**. На этапе детекции аномалией считается значимое отклонение факта `y(t)` от прогноза `ŷ(t)` (а также набор статистических правил). Поскольку предикторы — это другие датчики в тот же момент, модель улавливает нарушение **взаимной согласованности** сигналов.

Обучается по одной модели на каждый небулев датчик каждого ГПА (датчики `is_*` исключаются). На станции `ohangaron` это ~80+ моделей (ГПА-1/2/3).

## 3. Источник и подготовка данных

### 3.1 Источник
PostgreSQL, таблица `raw_data(datetime, point, value, health)` в long-формате (одна строка = один тег в момент времени). Загрузка — `PostgresDataLoader.fetch_training_data(cutoff_date)`.

### 3.2 Long → wide (`_to_wide`)
1. `datetime` → datetime, `value` → numeric.
2. **Округление к 5 минутам** (`round('5min')`) — единая сетка времени.
3. Дедупликация по `(datetime, point)`.
4. **Pivot:** строки = метки времени, столбцы = теги.
5. `ffill(limit=2)` — заполнение коротких пропусков (≤2 шага = 10 мин).

### 3.3 Маппинг тегов
`build_tag_mapping()` строит `point → feature` регулярками (`_normalize_tag`): `GPA-1.GPA-1.PD.PV → gas_pressure_out_gpa__GPA1` (а также `GT-n.GT-n.*`). Нераспознанные форматы пропускаются. Обратный маппинг `name_to_tag` сохраняется в `metadata.json`.

### 3.4 Отбор датчиков
В обучение идут столбцы с суффиксом `__GPA*`, **кроме** булевых `is_*` (статусы/режимы не моделируются регрессией). Затем `df_wide.ffill(limit=2)`.

## 4. Фильтрация простоев ГПА

Простой агрегата искажает «норму» (датчики константны, режим нерабочий), поэтому стоп-периоды исключаются **до** обучения по каждому ГПА:

1. Опорный сигнал оборотов: `rpm_tvd*` (или любой `rpm_*`).
2. Рабочий диапазон оборотов: `rng = quantile(rpm, 0.99) − quantile(rpm, 0.01)`.
3. **Стоп:** `rpm ≤ 0.05·rng` (обороты ниже 5% рабочего размаха).
4. **Прогрев:** к стоп-маске добавляются ещё 6 шагов (30 мин) после пуска — переходный процесс не считается нормой.
5. Стоп-строки → `NaN` и отбрасываются. Если рабочих точек получилось `< 100` — обучение идёт **без фильтра** (с предупреждением), чтобы не остаться без данных.

Побочный эффект: `sensor_range` и пороги детекции считаются по **рабочему** режиму.

## 5. Инжиниринг признаков (`_make_features`)

Для целевого датчика `target` формируются признаки:

| Группа | Признаки | Комментарий |
|--------|----------|-------------|
| Предикторы | все остальные датчики того же ГПА (что есть в данных) | контемпоральные (тот же момент `t`) |
| Лаги цели | `target_lag1`, `target_lag2`, `target_lag3` | значения 5/10/15 мин назад |
| Скользящие | `target_roll6_mean`, `target_roll6_std` | окно 6 шагов = 30 мин, **со сдвигом** `shift(1)` (только прошлое) |

После формирования — `dropna` по признакам и цели (убирает границы окон и пропуски). Если осталось `< 100` строк — датчик пропускается.

> Лаги/скользящие берутся строго из прошлого (`shift`), без утечки будущего. Контемпоральные предикторы — осознанный выбор (модель согласованности), но он имеет предел: одновременный отказ группы датчиков модель может «согласовать» (см. §11, T6).

## 6. Разбиение на обучение/валидацию

Временно́е (без перемешивания — корректно для рядов):

- `VAL_FRACTION = 0.2` → `n_val = max(50, 0.2·N)`.
- **Train:** первые `N − n_val` точек; **Validation:** последние `n_val` точек.
- Если после выделения валидации в train остаётся `< 50` точек — датчик пропускается.

Валидация используется для early stopping и для **честных** остаточных метрик (`residual_*_val`).

---

## 7. Модель и гиперпараметры

Алгоритм — **CatBoost `CatBoostRegressor`** с функцией потерь `RMSEWithUncertainty` (модель возвращает 2 выхода: среднее `ŷ` и дисперсию — основа доверительного коридора в детекции).

| Параметр | Значение | Зачем |
|----------|----------|-------|
| `loss_function` | `RMSEWithUncertainty` | прогноз + оценка неопределённости |
| `iterations` | `800` (затем `best_iter`) | верхний предел, реально — по early stopping |
| `depth` | `6` | умеренная глубина деревьев |
| `learning_rate` | `0.05` | устойчивое обучение |
| `l2_leaf_reg` | `20` | сильная L2-регуляризация (против переобучения) |
| `bootstrap_type` | `Bernoulli`, `subsample=0.8` | стохастичность, регуляризация |
| `random_seed` | `42` | воспроизводимость |
| `logging_level` | `Silent` | без шума в логах |
| `thread_count` | `cpu // n_jobs` | потоки на одну модель |

Глобальные константы детекции, сохраняемые в metadata: `ANOMALY_N_SIGMA = 5.0`, `MIN_BUFFER_PCT = 0.15`, `VAR_SMOOTHING = 24`.

## 8. Процедура обучения (двухэтапная)

Для каждого датчика (`train_single_sensor`):

**Этап 1 — подбор числа итераций (на train+val):**
1. `model_tmp = CatBoostRegressor(**CB_PARAMS)`.
2. `fit(X_tr, y_tr, eval_set=(X_val, y_val), early_stopping_rounds=50)`.
3. `best_iter = model_tmp.best_iteration_` (или 800).
4. По валидации: `val_resid = |y_val − ŷ_val|`, считаются `residual_std_val = std(val_resid)` и `residual_mean_val = mean(val_resid)` — **это фактически валидационная MAE**.

**Этап 2 — финальная модель (на 100% данных):**
5. `final_params = {**CB_PARAMS, 'iterations': best_iter}`.
6. `model.fit(X, y)` — дообучение на **всех** данных датчика с уже подобранным числом итераций (чтобы использовать максимум информации).
7. `y_mean = model.predict(X)[:,0]`, `r2 = r2_score(y, y_mean)` — **R² на обучающей выборке (in-sample)**.
8. `model.save_model(<sensor>.cbm)`.

> Нюанс (см. §11): финальный R² считается на 100% тех же данных, на которых обучались, → он оптимистичен. Честные остаточные метрики — это `residual_*_val` с этапа 1.

## 9. Что вычисляется и сохраняется

Метрики/параметры на модель (в `metadata.json → models[sensor]`):

| Поле | Смысл |
|------|-------|
| `r2_train` | R² на обучающей выборке (in-sample) |
| `residual_std_val` | σ абсолютных остатков на валидации |
| `residual_mean_val` | среднее абсолютных остатков на валидации (**= валидационная MAE**) |
| `best_iterations` | число деревьев после early stopping |
| `sensor_range` | рабочий размах: `percentile(y,99) − percentile(y,1)` |
| `predictors`, `feat_cols` | состав признаков |
| `tag`, `gpa_id`, `model_file` | привязка к SCADA-тегу/ГПА/файлу |

Глобально в `metadata.json`: `station_id`, `last_train_timestamp` (последняя метка обученных данных — граница «обучение/мониторинг»), `anomaly_n_sigma`, `min_buffer_pct`, `var_smoothing`, `gpa_ids`, `tag_to_name`, `name_to_tag`.

> ⚠️ Поле `mae` в API/UI **не заполняется** обучением (читается, но не пишется) → MAE везде `0.00`. Фактическая MAE существует как `residual_mean_val`. План (фаза 5) — заполнить и перейти на MAE.

## 10. Параллелизм и производительность

- Обучение распараллелено по датчикам через **joblib** (`Parallel`, backend `loky`), по каждому ГПА отдельным пакетом задач.
- `N_JOBS = max(1, cpu//2)` параллельных задач, `THREAD_COUNT = cpu // N_JOBS` потоков на модель — баланс «много моделей × многопоточный CatBoost».
- Прогресс и метрики по каждой модели печатаются в лог; считается суммарное время.

## 11. Использование обученного в детекции

Из `metadata.json` и `*.cbm` детектор (`live_predict.predict_sensor`) строит **доверительный коридор** вокруг прогноза:

1. `ŷ, y_var = model.predict(X)` (среднее и дисперсия).
2. `y_std = sqrt(сглаженная(y_var, окно=VAR_SMOOTHING=24))`, не ниже `residual_std_val`.
3. Полуширина коридора:
   `hw = max(n_sigma·y_std, sensor_range·MIN_BUFFER_PCT) · transient_boost`,
   где `n_sigma=5`, `transient_boost` расширяет коридор на быстрых переходах.
4. Нижний предел `hw` — `min_abs_error = sensor_range · min_abs_pct`, где `min_abs_pct` **зависит от R²** (≥0.95→0.10, ≥0.80→0.20, ≥0.50→0.30, иначе 0.50).
5. **ML-аномалия:** `|y − ŷ| > hw` при `is_live` (время > `last_train_timestamp`), с подавлением на стоянке/прогреве.

То есть метрики обучения напрямую управляют чувствительностью: `residual_std_val` задаёт пол шума, `sensor_range` — масштаб, `r2_train` — минимальный порог. (План фазы 5/7 переводит порог с R² на нормированную MAE и калибрует `n_sigma`.) Полное описание 7 детекторов — [06_anomaly_detection.md](06_anomaly_detection.md).

## 12. Запуск и воспроизводимость

```bat
cd backend
..\venv\Scripts\python.exe train_and_save_models.py --station ohangaron
:: до даты:           --cutoff-date 2026-06-15
:: проверка данных:   --dry-run     (число датчиков/точек без сохранения)
:: список станций:    --list-stations
```

- Детерминизм: `random_seed=42`; результат зависит от состава данных на момент запуска (`last_train_timestamp` фиксирует границу).
- Артефакты складываются в `models/<station>/` и используются детектором без изменений.
- Переобучение по расписанию **не предусмотрено** (обучение однократное «до сегодня»). Сигнал `model_drift` в `/api/health` — информационный.

## 13. Сильные стороны методики

- **Доменно осмысленно:** прогноз датчика по «соседям» + история ловит нарушения физической согласованности, а не просто выбросы.
- **Неопределённость из коробки:** `RMSEWithUncertainty` даёт адаптивный коридор (шире там, где модель менее уверена).
- **Защита от переобучения:** L2=20, subsample=0.8, early stopping, временна́я валидация.
- **Чистая «норма»:** фильтрация простоев и прогрева не даёт модели выучить нерабочие режимы.
- **Без утечки будущего** в лаг/скользящих признаках.
- **Масштабируемо:** независимые модели, параллельное обучение.

## 14. Ограничения и риски (с привязкой к плану)

| ID | Ограничение | Последствие | План |
|----|-------------|-------------|------|
| T1 | `mae` не заполняется (MAE=0 в UI) | оператор не видит ошибку модели в физ. единицах | фаза 5 |
| T2 | финальный R² — in-sample (на 100% данных) | завышен; переобучение не видно | фаза 5/6 |
| T3 | порог детекции зависит от оптимистичного R² | смещённая чувствительность | фаза 5 |

| T7 | бедные метаданные | нет размеров выборок, важности признаков, версии | фаза 6 |
| T8 | стейловые `.cbm` для `is_*` | мусорные файлы | фаза 6 |

1. **Метрика MAE вместо R² (фаза 5):** сохранять `mae_val`(=`residual_mean_val`), `rmse_val`, `nmae_val = mae_val/sensor_range`; заполнить контрактное поле `mae`; перевести порог детекции и UI на (нормированную) MAE.

3. **Робастное обучение (фаза 6):** отсев точек с большими остатками (k·MAD или trim 1%) перед финальным refit.

5. **Калибровка детекции и медленный дрейф (фаза 7):** измерять покрытие коридора и долю ложных; калибровать `n_sigma`; добавить отдельный трендовый (не-AR) детектор деградации.
