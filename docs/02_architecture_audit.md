# 02. Аудит архитектуры

Полный аудit архитектуры CS Monitor AI: сильные стороны, находки (со степенью серьёзности и рекомендацией), связь с планом доработок. Аудит обучения/детекции (метрики, пороги) — отдельно в [06_anomaly_detection.md](06_anomaly_detection.md) и в плане (фазы 5–7).

Легенда серьёзности: 🔴 высокая · 🟡 средняя · 🔵 низкая.

## 2.1 Сильные стороны

- **Чистое разделение фаз:** обучение (офлайн) ↔ детекция (онлайн-цикл) ↔ API (stateless-чтение) ↔ UI. Контракты простые.
- **Мульти-станционность** через YAML-конфиг и `StationConfig`; нет жёстких путей/имён в логике.
- **Устойчивость:** пул соединений + keepalive, single-instance lock, graceful shutdown, backoff, обработка `MemoryError`, авторестарт.
- **Производительность чтения:** даунсемплинг графиков в БД (≤1500 точек), mtime-кеш состояния, HTTP-кеш, кеш health-проверки БД, индексы.
- **Персистентность аномалий** в БД с дедупликацией — переживает рестарты предиктора.
- **Прагматичная доменная логика:** подавление ложных на остановленных ГПА, прогрев, переходные режимы, кросс-ГПА и смена режима.

## 2.2 Находки

### 🔴 Высокая

**A5. Нет аутентификации/авторизации на API.**
API открыт всем в сети; CORS разрешает только `localhost:*` (для прод-фронта с другого origin потребуется правка). Чтение телеметрии и аномалий не защищено.
*Рекомендация:* добавить аутентификацию (API-ключ/JWT/reverse-proxy с авторизацией) и сетевые ограничения (firewall/VPN) для продакшена; вынести список разрешённых origin в конфиг.

### 🟡 Средняя

**A1. Межпроцессное взаимодействие через JSON-файл состояния.**
`live_predict` пишет `state.json`, API его читает — связь через файловую систему. Файл содержит серии за 30 дней по всем датчикам (`v/p/lo/hi`), события, chart-аномалии → может быть многомегабайтным и полностью переписывается каждый цикл.
*Влияние:* рост размера, нагрузка на диск, сложность горизонтального масштабирования (несколько хостов API не увидят локальный файл).
*Рекомендация:* среднесрочно — вынести оперативное состояние в БД или кэш (Redis); серии графиков уже могут строиться напрямую из `raw_data` (API это умеет в `/chart`), что снизит зависимость от файла. Минимум — ограничить объём (фаза 8 плана срезает `p/lo/hi` на истории).

**A2. Два процесса без супервизора.**
API (4 воркера uvicorn) и предиктор стартуют двумя `.bat` с самостоятельными циклами рестарта. Нет единой оркестрации, общего health, управляемого порядка старта (миграции → модели → сервисы).
*Рекомендация:* единый запуск `run_system.py` (**план, фаза 12**) + системный сервис (NSSM/Task Scheduler/systemd) поверх него.

**A3. Дублирование UI.**
В `live_predict.py` встроен полноценный HTML-дашборд (`build_dashboard`, ~1800 строк HTML/CSS/JS) — и параллельно существует React-фронт (`frontend/`). Две презентации дублируют логику (health-score, heatmap, журнал, спарклайны).
*Влияние:* двойная поддержка, рассинхронизация UI/логики.
*Рекомендация:* оставить один UI (React как основной), встроенный HTML — убрать или вынести в отдельный опциональный рендер.

**A4. Монолит `live_predict.py` (~2716 строк).**
В одном файле: загрузка моделей, признаки, 7 детекторов, кросс-ГПА и смена режима, запись состояния, генерация HTML, главный цикл.
*Влияние:* трудно тестировать и менять без регрессий.
*Рекомендация:* разнести по модулям: `detectors/`, `state_writer.py`, `dashboard.py`, `loop.py`. Покрыть детекторы юнит-тестами (см. A10).

**A6. Разрозненная работа с часовыми поясами.**
Конверсии UTC↔`Etc/GMT-5` повторяются в `main.py`, `live_predict.py`, `data_loader.py`. Для будущей записи `health`/журнала по ключу `(point, datetime)` это критично (легко рассинхронить).
*Рекомендация:* централизовать TZ-утилиты (один модуль), покрыть тестами round-trip.

**A8. Нет фреймворка миграций БД.**
Схема создаётся ad-hoc: `ensure_anomalies_table()`, `fix_anomalies_table.py`, `ensure_indexes.py`. Версионирования схемы нет.
*Рекомендация:* при росте схемы — Alembic или хотя бы единый версионируемый `migrate_db.py` (**план, фаза 1**) с журналом применённых миграций.

**A9. Память предиктора.**
Wide-pivot всех тегов за 30 дней может быть тяжёлым. Есть `MAX_HISTORY_DAYS`, тримминг и обработка `MemoryError` (сброс) — но это «спасательный круг», а не управление ресурсами.
*Рекомендация:* профилировать пик памяти при текущем числе тегов; при росте — обрабатывать по ГПА/частями, хранить только нужные окна.

**A10. Тонкое тестовое покрытие.**
Только smoke-тесты (импорты, словарь, lock, валидация station_id, severity). Нет тестов детекторов, контракта API, качества данных; ML-корректность не проверяется.
*Рекомендация:* фикстуры синтетических рядов с заведомыми аномалиями; юнит-тесты детекторов; контрактные тесты API; интеграционный прогон на тест-БД (**план, фаза 13**).

**A12. Ограниченная наблюдаемость.**
Есть логи (ротация), `/api/health` (ok/degraded/down, возраст state, доступность БД), `model_drift`. Нет метрик (Prometheus) и алертинга (журнал уведомлений добавляется планом).
*Рекомендация:* экспортировать метрики (latency API, длительность цикла, число аномалий, лаг данных) и настроить алерты на `degraded/down` и лаг данных.

**A13. Хрупкий маппинг тегов и отсутствие валидации ingest.**
`_normalize_tag` парсит теги регулярками (`GPA-n.GPA-n.*`, `GT-n.GT-n.*`); нераспознанные форматы тихо пропускаются (debug-лог). Данные в `raw_data` приходят извне без валидации.
*Рекомендация:* отчёт покрытия тегов (сколько распознано/пропущено по station), алерт при появлении неизвестных тегов; базовая валидация диапазонов на ingest.

### 🔵 Низкая

**A7. «Глотание» ошибок.** Широкие `try/except` повышают устойчивость, но часть веток молча подменяет данные (fallback chart-аномалий, чтение state). *Рекомендация:* логировать все подмены на WARNING.

**A11. Масштабируемость на много станций.** Предиктор — на станцию (`--station`) → N процессов; пул на воркер (4×10) + предиктор создают давление на БД. *Рекомендация:* следить за `max_connections`, при росте — pgbouncer; рассмотреть один многостанционный предиктор.

**A14. Привязка фронта к станции по умолчанию.** Жёсткий `DEFAULT_STATION='ohangaron'` и набор compat-алиасов `/api/sensors`, `/api/stats`… — технодолг. *Рекомендация:* постепенно перевести фронт на `/api/stations/{id}/...` и убрать алиасы.

## 2.4 Приоритеты

1. **Безопасность (A5)** — до вывода в сеть общего пользования.
2. **Единый запуск + супервизор (A2)** — операционная надёжность (план, фаза 12).
3. **Тесты (A10)** — перед внедрением записей в `health`/журнал, чтобы не словить регрессии.
4. Остальное (A1/A3/A4/A6/A8/A9/A12/A13) — поэтапно как техдолг.
