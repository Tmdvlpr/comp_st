# Методология обучения моделей и детекции аномалий — CS Monitor AI (v23)

> Промпт-описание системы. Самодостаточно: даёт полную картину того, КАК система учится и КАК
> ловит аномалии на газоперекачивающих агрегатах (ГПА). Термины — как в коде (файлы указаны).

---

## 0. Что это и зачем

Система непрерывного мониторинга **48 датчиков × 3 ГПА** (давления, температуры, обороты,
вибрация, осевой сдвиг, политропный напор, температуры масла/подшипников). Телеметрия ~5-минутная.
Задача: строить для каждого датчика **коридор нормы** и выявлять **аномалии** трёх разных природ —
от точечного выброса до медленного структурного сдвига состояния машины.

Базовый принцип: **аномалия = расхождение факта с тем, что модель считает нормой для текущих
условий**, с честной статистической гарантией покрытия и защитой от «заглатывания» деградации.

Конвенции: БД хранит время в UTC; pandas-индекс и метаданные — `Etc/GMT-5`-naive. Широкий формат
колонок `<sensor>__GPA<n>`. Все производные записи в БД конвертируются в UTC.

## 1. Данные и признаки (`train.py`, `data_loader.py`, `weather.py`)

- **Очистка/маски:** `physically_clean` (физические границы из `methodology.limits`),
  `verified_healthy_mask` (точка «здорова»), `steady_running_mask` (агрегат работает и в
  установившемся режиме), `label_regime` (steady / stop / transition / warmup).
- **Режимы (Mondrian-таксономия):** `regime_key = steady × submode(mainline|ring) × бин нагрузки`.
  Бины нагрузки — квантильная нарезка (до 3) по первой доступной оси из
  `(rpm_tvd, pressure_ratio, rpm_tvd_red, fuel_gas_flow)`; границы сохраняются и переиспользуются в live.
- **Доменные признаки** (`add_domain_features_gpa`): приведённые обороты (`rpm_*_red`, ISO 3977-2 /
  PTC 22), политропный напор, shaft-mismatch, `ambient_temp` (из `weather.py`), подход к помпажу
  (`avo_approach`), время/счётчики от пуска.

## 2. Обучение модели одного датчика (`train.py: train_sensor`, `train_sensor_pooled`)

- **Модель:** для КАЖДОГО датчика **CatBoost** предсказывает его значение по ДРУГИМ датчикам +
  доменным признакам (кросс-сенсорная регрессия). Датчик не предсказывает сам себя.
- **Голова неопределённости:** `RMSEWithUncertainty` + posterior sampling (SGLB) +
  virtual ensembles. Единая функция `ensemble_sigma_uepi(mdl, X, n_trees)` возвращает:
  - `mean` — предикт;
  - `u_epi` — эпистемическая (knowledge) неопределённость = разброс виртуального ансамбля («модель
    в незнакомой зоне»);
  - `u_ale` — алеаторная (data) неопределённость = шум данных;
  - `σ = √(u_epi + u_ale)`.
  **КРИТИЧНО:** эта же функция используется и в обучении, и в `live_predict` — иначе гарантия
  покрытия коридора невалидна.
- **Pooled vs per-unit:** термо/механика — одна **кросс-ГПА** модель на z-нормированных данных
  (перенос между агрегатами, `_PooledAdapter`); **вибрация — per-unit** (механическая подпись
  агрегата уникальна). Маршрутизация — `is_pooled_target` (config `methodology.pooling`).
- **Окно обучения** `[from_date, cutoff]`, ранняя остановка по валидации.

## 3. Калибровка коридора — block-Mondrian split-conformal (`train.py`)

Коридор строится на **healthy-steady точках СТРОГО после cutoff** (held-out по времени), по режимам.

- **Нонконформность (два скора на одних точках, хранятся оба — `mode="dual"`):**
  - `s_norm = |факт − предикт| / σ`  → порог `q̂_norm` → **hybrid**-коридор `предикт ± q̂_norm·σ`
    (раздувается там, где модель не уверена);
  - `s_abs = |факт − предикт|`       → порог `q̂_abs`  → **conformal**-коридор `предикт ± q̂_abs`
    (плоский).
- **Порог, устойчивый к автокорреляции:** moving-block bootstrap (длина блока = первый лаг с
  ACF < 1/e), `n_eff = n // block_len`; наивный split-квантиль `⌈(n+1)(1−α)⌉`; `α = 0.02` (цель
  покрытия 98%). `conservative_q` = перцентиль bootstrap-распределения квантиля.
- **Гейты качества:**
  1. модельный `beats_baseline` (MAE_eval < MAE режимного среднего) → `detector_mode=ml_corridor`,
     иначе `univariate_only`;
  2. статистический `n_eff ≥ N_EFF_MIN(=49)` на режим (иначе режим → univariate).
- **`corridor_quality`:** `genuine` (R²≥0.3) vs `steady_band` (R²<0.3, полоса «нормального
  диапазона», но не предсказательная).

### 3.1 Спектр предикторов (расширение коридора на «трудные» датчики)
split-conformal **инвариантен к выбору предиктора** (при обмениваемости покрытие держится для
любого ŷ). Поэтому предиктор — свободный параметр на спектре:
- **CatBoost-остаток** (по умолчанию, где модель предсказательна);
- **self-conformal** (`CS_SELF_BAND`): предиктор = healthy-медиана режима, скор `|значение−центр|`
  через ТОТ ЖЕ калибратор — для датчиков с `R²<0` (напр. вибрация), где кросс-сенсорная регрессия
  бесполезна, но healthy-распределение значения концентрировано;
- **FORCE** (`CS_FORCE_CORRIDOR`): снять гейт beats-baseline и строить коридор ВЕЗДЕ (при `n_eff<49`
  — консервативный раздув ×1.5). Даёт conformal+hybrid на всех 48.

### 3.2 Персональный CV выбора режима (`_interval_cv.py`) — движок Фазы 2
Для КАЖДОГО датчика **rolling-origin блочный CV** честно выбирает лучший режим интервала из
`{conformal | hybrid | self}` по **coverage-efficiency**:
- хронологические блоки; калибровка на ранних → выбор режима на forward-валид-фолдах → отчётное
  покрытие на ИЗОЛИРОВАННОМ последнем блоке (нет winner's curse; self-центр считается на калиб-фолде,
  без утечки);
- правило выбора: покрытие ближе к `1−α`, недобор штрафуется ×3; анти-winner's-curse держит
  стабильный `conformal`, только если он сам ≥ цели;
- флаги: `SUBTARGET` (лучший режим < 0.95 → полосу НЕ навязываем, честный `univariate`);
  `LEVEL_SHIFT` (уровень уехал > 4σ за окно → полоса перецентрирована, пометить на проверку);
- результат — per-sensor `corridor_mode` в metadata (`_apply_cv_selection.py`), который читает live.

Эмпирика (прод-модели, forward-CV, 48 датчиков): глобально conformal 0.925 → per-sensor
best conf/hybr **0.946** → + self-band **0.985** (44/48 ≥ 0.95).

## 4. Детекция аномалий — 3 комплементарных детектора на ЗАМОРОЖЕННОЙ healthy-базе

Ключ к anomaly-safety: **все эталоны заморожены** на обучающем healthy-распределении и НЕ
пересчитываются на свежих данных (иначе детектор «уезжает» вместе с деградацией — «варка лягушки»).
**Онлайн-адаптации порога нет** (train-once; рекалибровка только офлайн-переобучением).

### Д1 — Коридор (точечная детекция, `live_predict.py`)
Факт вне per-sensor полосы (`conformal`/`hybrid` по выбранному `corridor_mode`;
приоритет env `CS_CORRIDOR_MODE` > per-sensor > станция > metadata > 'conformal'). Пробой становится
аномалией при `resid > 0.25·sensor_range`, на live-точке, при работающем агрегате; для стохастических
(вибро/осевой/сдвиг) — плюс **персистенция K-из-N** (3 из 5). `corridor_valid` гасит полосу на
маргинальной экстраполяции/OOD.

### Д2 — Эпистемическая новизна (OOD, `live_predict.py`)
`u_epi > κ = know_p95 × 1.5` **ИЛИ** маргинальность (входной признак вне train-диапазона ±10%) →
«модель в незнакомом режиме». Это отдельная ось: σ её частично ловит (hybrid раздувается), но
маргинальность — независимый флаг.

### Д3 — Структурный дрейф (СИСТЕМНЫЙ, `system_monitor.py` + `drift_monitor.py`) — НОВОЕ
Коридор ловит выброс ТОЧКИ, но медленный сдвиг режима (уровень уехал + связи датчиков сломались)
он пропускает: после перецентровки точки снова «внутри». Ловит это Д3:
- **`system_monitor.py` (unit-level, ГЛАВНЫЙ):** PCA на замороженной healthy-базе агрегата (все
  датчики совместно, z-норма) → **SPE** (Q, остаток вне здорового подпространства = слом структуры
  связей / концепт-сдвиг) + **T²** (махаланобис внутри). Контрольный предел = 99-й перцентиль базы.
  Индекс = доля свежих точек за пределом + медианное превышение + вклад датчиков + флот-контраст
  (unit-специфичное vs сезонное). Один защитимый индекс на агрегат: «машина структурно в ином
  состоянии, чем её healthy-база».
- **`drift_monitor.py` (per-sensor drill-down):** сдвиг уровня (в σ), слом корреляций датчика с
  соседями (смена знака), разброс; флот-относительность (`UNIT_OUTLIER` vs `FLEET_COMMON`). Порог
  сдвига уровня — **4σ** (`TAU_LEVEL`), корреляционный слом — по Δкорр и числу смен знака.
- **Иерархия:** агрегат (SPE/T²) → датчик (drift) → точка (коридор + CV).
- **Дискриминатор** «дефект vs сезон»: t-статистика тренда бесполезна (огромна на любом сезонном
  дрейфе при больших n) → используем **сдвиг уровня в σ** + **флот-контраст** (тот же сезонный
  переход у всех ГПА; уехал только один → unit-специфично).

## 5. Прод-цикл (`run.py` → `run_system.py`)

`python run.py` поднимает фронт (Vite) + бэк. `run_system.py`:
1. миграции БД + индексы;
2. FastAPI (:8000);
3. **онлайн-ML** `live_predict.run_continuous` — реальный 5-мин цикл: **читает БД → считает на лету →
   пишет новое в БД** (`predictions` — серия p/lo/hi/lo2/hi2/e; `anomalies_t` — эпизоды + SHAP;
   `domain`, `raw_data.health`). На старте `catch_up_missing` догоняет пропуск оконными запросами;
4. **daemon-поток здоровья** (`_health_loop`, раз в сутки, изолированные подпроцессы):
   `system_monitor.py --write-db` → таблица **`system_health_t`** (unit-level SPE/T²/вердикт/вклад);
   `drift_monitor.py` → JSON drill-down. Тяжёлый PCA не мешает 5-мин ML-циклу.

Гард записи в БД: `CS_DISABLE_DB_WRITE` (read-only в тестах/офлайн-анализе; в проде мониторы пишут).

## 6. Открытые границы (честно)

- Голодная калибровка стоящего агрегата (ГПА-3 в standby): рабочий коридор честно недостижим, пока
  агрегат не запустят и не наберёт свежих working-steady точек; до того — `univariate`/провизорный
  донор с task-level гарантией.
- Гарантии формулируются как «≥ 1−α минус измеримый зазор», не как точные значения.
- Д3 SPE ловит «структурно иное, чем healthy-база» = концепт-сдвиг; дефект vs доброкачественная
  смена режима — за инженером, но флот-контраст исключает «просто сезон».
